# 好文示例

这些片段示范目标写作风格，供 Claude 对照校准。

## 示例 1 — 技术总结型：结论先行，有判断力

> **场景：** 解释为什么不建议在团队项目里用 Git rebase

---

简单说结论：rebase 会重写历史，在共享分支上用就是在制造麻烦。

这不是观点，是事实。你把一个 commit 的 parent 换掉之后，那个 commit 的哈希就变了。别人基于旧哈希拉下来的东西，和你现在推上去的东西，在 Git 看来是两条完全不同的历史线。合并的时候会出现重复 commit，强推会直接丢别人的工作。

实践里我见过最常见的场景是这样的：A 在 feature 分支做了 rebase，B 没拉最新，继续在旧的基础上提交，push 的时候发现冲突，解决完之后 log 里出现了两份一样的变更。没人知道删哪个，最后两个都留着。

所以问题不是 rebase 本身不好，而是它适合哪个场景。本地整理自己的提交，或者 pull 时用 rebase 保持线性历史，都 OK。一旦涉及多人共享的分支，merge 是更安全的默认选项，代价是历史稍微不那么干净。这个代价值得付。

---

**为什么是好文：**
- 第一句直接给结论，不铺垫
- 解释机制（哈希变化），不只说"这样不好"
- 举了一个具体场景，不是抽象举例
- 末尾给出判断边界，不是一刀切

## 示例 2 — 职场反思型：从具体情境出发，带价值判断

> **场景：** 关于"无效会议"这个老话题

---

我最近开始在日历上标"拒绝会议"的时间段，大概每天两小时。不是因为会议本身不重要，而是因为我发现自己在大量会议里做的事情，其实是在帮别人拿本来应该自己拿的决定。

这是一种挺隐蔽的成本。表面上你参与了，有贡献，但实际上你把自己的专注时间兑换成了别人的决策效率。对方省了时间，你的深度工作窗口被切碎了。

不是所有会议都这样。对齐方向、同步状态、做真正需要多人判断的决策，都值得开。问题出在"找个人来确认一下"和"我不确定，叫他来一起想想"这类会议。这类会议的本质是外包思考。

所以我现在接会议之前会先问一个问题：这件事如果我不在场，会议能进行吗？如果能，我大概不需要去。

---

**为什么是好文：**
- 从具体行为（标时间段）切入，不从"会议问题"开始
- 分析出成本的本质（专注时间换决策效率），有判断力
- 区分"值得开"和"不值得开"，有边界感
- 结尾给出可操作的判断标准

## 示例 3 — 故障复盘型：症状→猜测→验证→结论

> **场景：** 记录一次线上服务 OOM 排查

---

结论先说：是 Kafka consumer 没有限制 fetch 大小，把一个 200MB 的消息拉进来撑爆了堆。

当时的现象是服务每隔几小时就重启一次，没有明显的流量高峰，GC 日志里看不出 Full GC 的规律。第一反应是内存泄漏，加了 -Xmx 之后问题消失了几天，以为解决了，结果又来。

真正的突破口是一条 Kafka broker 的日志，显示某个 partition 里有条消息比正常消息大了三个数量级。拉出来看，是上游服务做了一次批量导出，把几万条记录序列化成一个事件发出来，200MB。

消费端没有设 `max.partition.fetch.bytes`，默认值是 1MB，但 broker 端 `message.max.bytes` 调大过，所以消息发出去了，消费端一次拉取就把整个 200MB 装进内存。

修复是两步：限制 consumer 的 fetch 大小，同时在上游加批次大小校验，拦住超大事件。加 -Xmx 那步是假修复，掩盖了问题。

---

**为什么是好文：**
- 结论在第一句
- 按时间线重建排查过程，有症状、有误判、有突破口
- 解释了根因的具体机制，不是"内存不够"这种模糊结论
- 明确指出假修复，提醒读者注意同类陷阱
