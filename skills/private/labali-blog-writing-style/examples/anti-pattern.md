# 反例：需要避免的写作模式

这些片段示范常见的质量问题，供 Claude 识别并纠正。

---

## 反例 1 — AI 腔：过度抽象，缺乏具体判断

> **原文（有问题的）**

在当今快速变化的技术环境中，我们需要不断提升自己的能力，以应对各种挑战。技术的发展日新月异，只有保持学习的热情，才能在竞争中立于不败之地。作为工程师，我们应该拥抱变化，持续成长，不断突破自我的边界。

**问题：**
- 全是抽象概念，没有一句话说了"什么挑战"、"怎么应对"
- "日新月异"、"立于不败之地"是通用励志句，任何行业都能套
- 读者读完不知道作者的实际经历和判断是什么
- 这段话可以出现在任何人的任何文章里

**修改方向：** 把"我们"换成"我"，把抽象换成一个具体的事情，说清楚你在什么情况下意识到了什么，做了什么选择，结果怎样。

## 反例 2 — 教科书腔：定义堆砌，没有作者视角

> **原文（有问题的）**

Git rebase 是一种 Git 操作，用于将一个分支的提交重新应用到另一个分支上。它的工作原理是找到两个分支的共同祖先，然后将当前分支上的每个提交依次应用到目标分支的顶端。与 merge 相比，rebase 可以产生更线性的提交历史。

**问题：**
- 这是 Git 文档里的解释，不是作者的经验
- 没有判断：什么时候用？什么时候不该用？作者的实际经历是什么？
- 读者去看官方文档能学到同样的东西
- 没有"为什么这件事值得写一篇文章"的答案

**修改方向：** 从作者自己踩过的坑出发，解释清楚一个你真正理解的场景，带出自己的判断。

## 反例 3 — 列表滥用：把应该是段落的内容拆成碎片

> **原文（有问题的）**

关于代码审查，我有以下几点想法：

- 代码审查很重要
- 要认真对待每一个 PR
- 不要只关注语法错误
- 要看整体设计
- 要给出有建设性的反馈
- 要尊重提交者的劳动
- 要及时回复

**问题：**
- 每个 bullet 都是孤立的断言，没有任何解释
- 这些观点是常识，读者已经知道
- 用列表结构掩盖了"作者没有实质内容"的事实
- 拆成列表之后，每个点都太轻，没有一个值得认真读

**修改方向：** 选其中一到两个观点，认真展开：为什么？在什么情况下？你是怎么做的？效果如何？

## 反例 4 — 强行升华：结尾贴"大道理"标签

> **原文（有问题的）**

...总的来说，这次排查花了我两天时间，找到了 Kafka fetch size 的问题。

技术的道路是无止境的。每一次故障都是一次成长的机会。只要我们保持谦逊、持续学习、勇于面对困难，就一定能成为更好的工程师。希望这篇文章能对大家有所帮助。

**问题：**
- 前面的排查过程是有价值的，这个结尾把它拉低了
- "技术的道路是无止境的"和这篇文章毫无关系
- "希望这篇文章能对大家有所帮助"是废话，写了就是在浪费读者时间
- 用通用励志句收尾是缺乏真实结论的表现

**修改方向：** 结尾应该回到文章本身的判断。这次排查教会你什么具体的东西？下次遇到类似问题你会先看哪里？这才是有价值的收尾。

## 反例 5 — 中性腔：刻意回避判断

> **原文（有问题的）**

关于是否应该在工作中使用 AI 辅助工具，不同的人有不同的看法。有些人认为 AI 工具可以提升效率，有些人担心会影响思维能力的培养。这两种观点都有其合理之处，最终还是要根据个人情况和具体场景来决定。

**问题：**
- "两种观点都有其合理之处"等于什么都没说
- 作者明显有意见，但不愿意表达
- 这种写法通常是怕"说错了被批评"的防御性写作
- 读者来看这篇文章，是想知道作者怎么想的

**修改方向：** 直接说你的判断。你在什么情况下用？什么情况下不用？你遇到过什么具体问题？把你的视角摆出来，读者自己会判断是否认同。
